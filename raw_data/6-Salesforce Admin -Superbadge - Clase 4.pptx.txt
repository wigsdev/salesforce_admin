Superbadges: La Prueba de Fuego del Administrador
¿Qué es un Superbadge?
No es un módulo normal. Es una Credencial basada en Habilidades.
Adiós a las instrucciones: Aquí no existe el "Paso 1, haz clic aquí". Salesforce te da un Caso de Negocio (un problema real de una empresa ficticia) y tú debes decidir cómo resolverlo.
Evaluación Automática: Un "robot" entra a tu organización de Salesforce y verifica si la configuración funciona correctamente según los requisitos del cliente.

¿Por qué son vitales para tu carrera?
El Puente Teórico-Práctico: Los módulos te enseñan a usar el martillo; los Superbadges te piden construir una casa.
Resume Booster (CV): Los reclutadores saben que un Superbadge no se consigue leyendo, se consigue haciendo. Valen mucho más que 100 badges normales.
Prerrequisito de Certificación: Para algunas certificaciones avanzadas, debes tener ciertos Superbadges completados.

Object Relationships Superbadge Unit
Dominando la Arquitectura de Datos: Object Relationships Superbadge Unit
1. ¿De qué se trata? (El Escenario)
Contexto de Negocio: Trabajaremos para "Tenor Management", una agencia que gestiona bandas de rock (como "Flow & the Low Codes").
El Reto: Necesitan una App llamada "World Tour Manager" para organizar sus giras mundiales.
Tu Misión: Dejar de usar hojas de cálculo y construir una arquitectura relacional en Salesforce que conecte Bandas, Álbumes, Canciones, Tours y Recintos (Venues).

¿Por qué es importante?
Dominando la Arquitectura de Datos: Object Relationships Superbadge Unit
Es la base de todo: Si el modelo de datos está mal, la seguridad, los reportes y las automatizaciones fallarán.
Validación de Habilidad Crítica: Demuestra que sabes decidir cuándo usar una relación fuerte (Master-Detail) o una débil (Lookup) y cómo impacta eso en el borrado de datos.
Pensamiento de Arquitecto: Te obliga a pensar en "cascada" (cómo fluyen los datos de un objeto a otro).

Objetivo Principal
Diseñar e Implementar un Modelo de Datos Escalable:
Usar objetos personalizados (Song, Album, Track List).
Configurar objetos estándar (Account como Venues, Campaign como Tours).
Establecer las relaciones correctas para que la información no quede aislada.
Entregable: Una App funcional donde el manager pueda ver qué canciones se tocaron en qué concierto y en qué recinto.

 Conceptos Teóricos Necesarios (Prerrequisitos)
Tipos de Relaciones:
Master-Detail (Maestro-Detalle): Para datos que dependen totalmente del padre (ej. Canciones dentro de un Álbum).
Lookup (Búsqueda): Para datos relacionados pero independientes (ej. Un Venue relacionado a un Tour).
Junction Objects (Muchos a Muchos): Entender cómo conectar dos padres (ej. Un Artista puede tener muchas Canciones, y una Canción tener muchos Artistas).
Schema Builder: Saber usar el lienzo visual para crear y verificar relaciones rápidamente.
Lightning App Builder: Configurar visibilidad de componentes basada en filtros de campos relacionados.

Travel Approval App
El Problema de Negocio
Actualmente, las aprobaciones de viaje se hacen por email.
Problema: No hay historial, no se pueden sacar reportes de "Gasto por Departamento", y los correos se pierden.
Solución: Crear una App dedicada con objetos estructurados.
La Arquitectura Visualicen esto:
Objeto Department (Departamento): Actuará como un catálogo maestro (IT, Ventas, HR).
Objeto Travel Approval (Solicitud de Viaje): Es el objeto transaccional. Aquí vive la acción.
La Relación: Cada Solicitud pertenece a un Departamento.

Travel Approval App
Truco de Experto: "Auto-Number" vs "Text"
Cuando creen el objeto Travel Approval, Salesforce les preguntará por el "Record Name".
El Error Común: Dejarlo como Texto. (El usuario tendrá que inventar un nombre como "Viaje a NY").
El Consejo Pro: Usen Auto-Number (Ej: TA-{0000}).
Por qué: Evita duplicados ("Viaje a NY" vs "Viaje a New York") y le da una referencia única profesional a cada solicitud.

Objetos, Tabs y la "Invisibilidad"
1. La Regla de los 3 Pasos  Para que un usuario vea lo que crearon, necesitan 3 cosas:
El Objeto: La base de datos (El esqueleto).
La Pestaña (Tab): La puerta de entrada. Sin Tab, el objeto existe pero es invisible en la barra de navegación.
La App: La colección de pestañas. Deben agregar sus Tabs a la "Travel App".
2. Tipos de Campos Clave en este Proyecto
Currency (Moneda): Para el costo del viaje. (Recuerden: Salesforce maneja la simbología automática).
Date (Fecha): Inicio y Fin. (Vital para validaciones futuras: Fecha Fin no puede ser antes de Fecha Inicio).
Checkbox: "¿Fuera del Estado?". Simple pero poderoso para filtros rápidos en reportes.
3. List Views (El olvido frecuente)
Una vez creado el objeto, la vista por defecto es "Recently Viewed" (Vistos Recientemente), que estará VACÍA y parecerá que se rompió.
Truco: Enseñen a crear inmediatamente la vista "All" (Todos) para que no entren en pánico al no ver sus registros.

Schema Builder vs. Object Manager
1. Object Manager (El camino lento y seguro)
Es donde crearán la mayoría de los campos paso a paso.
Ideal para configurar detalles finos (descripciones, textos de ayuda).
2. Schema Builder (El camino rápido y visual)
En este proyecto, prueben usar el Schema Builder.
Ventaja: Pueden crear el objeto Department y Travel Approval y arrastrar la línea de relación entre ellos visualmente.
Advertencia de Oro: Cuando crean campos desde el Schema Builder, NO se agregan automáticamente al Page Layout.
Consecuencia: Crean el campo, van al registro... ¡y no está! Tienen que ir al Page Layout y arrastrarlo manualmente.

Aprendizaje para Proyecto Final
APRENDIZAJE
1. Arquitectura Relacional (El cimiento del edificio)
Objetivo: Entender que los objetos no son islas.
Impacto en el Proyecto: Si en el Sprint 1 definen mal la relación (ej. usan Lookup cuando debió ser Master-Detail), en el Sprint 3 no podrán crear los Reportes que pide el cliente ni los Roll-Up Summaries.
Lección: Un error en el modelo de datos se paga con horas de reconstrucción en el futuro.
2. Experiencia de Usuario "Navegable" (UX)
Objetivo: Comprender que el usuario final no entra al "Setup".
Impacto en el Proyecto: De nada sirve crear la mejor automatización si el usuario no tiene una App (Conjunto de pestañas) ordenada o si el campo clave está escondido al final del formulario.
Lección: "Si no es intuitivo, el usuario dirá que el sistema no sirve."
3. Preparación para la IA (Agentforce Ready)
Objetivo: Estructura de datos limpia y tipificada.
Impacto en el Proyecto: En el Mes 3/4 usaremos Agentes de IA. Un Agente puede leer un campo de tipo "Moneda" o "Fecha", pero se confunde con campos de "Texto" mal usados.
Lección: Usar el tipo de dato correcto hoy (Currency, Date, Picklist) asegura que la IA pueda trabajar mañana.
